# Product Goals

## 목적

이 템플릿은 Next.js 기반 프로젝트의 빠른 부트스트래핑을 위한 공용 베이스 아키텍처입니다. 프론트엔드 개발 생산성을 극대화하기 위한 목표는 다음과 같습니다.

- 신규 프로젝트를 `복사` 또는 `Fork`하여 즉시 메인 비즈니스 로직 개발을 시작할 수 있어야 합니다.
- 인증, 권한, 다국어 처리(i18n), 전역 상태 관리, 공통 레이아웃 구조, 백엔드 연동(BaaS), 테스트 환경 체계를 초기부터 일관되게 제공해야 합니다.
- 팀 내 개발자들이 초기 프로젝트 세팅 시 발생하는 보일러플레이트 중복 작업을 제거해야 합니다.
- 서버 사이드 렌더링(SSR) 중심의 App Router 구조를 강제하여 초기 렌더링(First Load) 퍼포먼스를 확보하고, 상태 관리나 상호작용이 필요한 부분만 클라이언트 컴포넌트로 분리하는 점진적 하이드레이션(Progressive Hydration)을 채택합니다.

## 템플릿의 범위

이번 단계에서 정의할 아키텍처 범위:

- Next.js App Router 기반의 라우팅 매커니즘
- TypeScript 엄격모드 설정 및 타입 가이던스
- Tailwind CSS + shadcn/ui 기반의 디자인 시스템 및 토큰 구조
- 다국어(i18n) 설정 및 라우팅 병합 구조
- Supabase Auth를 활용한 인증/보안 시스템
- 테스트 전략 (`Vitest` 기반 단위 테스트 및 `Playwright` 기반 E2E 테스트)
- 모바일/데스크톱 대응 가능한 반응형 공통 셸(App Shell) 레이아웃 전략
- 도메인 주도 설계(DDD) 관점의 향후 기능 플러그인 확장을 대비한 폴더 트리

추후 구현하거나 정책으로 분리할 설계 범위:

- `echarts` 차트 활용 플러그인
- Complex Data Table 플러그인 구현
- 실제 프로덕션 수준의 멀티 테넌트(Multi-tenant) 아키텍처
- RBAC(Role Based Access Control) 기반 세분화된 권한 통제

## 성공 기준

- 템플릿 생성 후 `.env` 등 최소한의 환경변수만으로 로컬 구동 시 에러가 나지 않아야 합니다.
- 인증, 다국어, 레이아웃 변경, API Fetching 방식이 프론트엔드 개발자에게 직관적이어야 합니다.
- 서버 컴포넌트(RSC) 사용에 대한 기준이 코드단에 명확하게 나뉘어 있어야 하며, `use client` 남용을 막습니다.
- 실제 운영 환경(Production)으로 스케일 업 할 때, 구조를 대폭 수정하지 않고 모듈(Feature) 추가만으로 대응할 수 있어야 합니다.

## 템플릿 기본 원칙

- 파일/폴더 라우팅은 철저히 Next.js App Router 디렉터리 구조를 따릅니다.
- 데이터 Fetching과 Layout의 기본값은 RSC(React Server Component)입니다.
- 단, 사용자의 상호작용(onClick 등), Web APIs 접근(`window`, `document`), React Context나 Zustand 스토어 구독이 필요한 영역에서만 클라이언트 컴포넌트로 전환합니다.
- 공통 유틸 및 UI 요소는 재사용 가능하도록 완전히 추상화하여 분리합니다.
- 플러그인(차트 등)은 메인 번들 사이즈를 위협하지 않는 선에서 동적 임포트(Dynamic Import) 또는 배제된 채로 추가/삭제가 가능해야 합니다.
