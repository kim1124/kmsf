Chapter 4 / 10

cli.ts의 흐름

한 줄 요약

runCli()는 5단계로 일한다: ① 인자 파싱 → ② --help/--version 처리 → ③ logger 준비 → ④ 누락된 옵션을 prompts로 받음 → ⑤ scaffold() 호출 + 결과 출력. 모든 에러는 catch해서 적절한 exit code로 변환한다.

4.1 runCli()의 큰 골격

코드는 길지만 핵심 흐름은 간단합니다. 한 화면에 압축해 보면:

① parseCliArgs(argv)
CLI 인자를 객체로 파싱 (args.ts)
② --help / --version 빠른 종료
메시지 출력 후 exit 0
③ logger + banner 준비
색상 / 로그 레벨 결정 (logger.ts, banner.ts)
④ resolveScaffoldOptions(args)
누락된 옵션은 prompts로 사용자에게 질문 (prompts.ts)
⑤ scaffold(options)
실제 파일 생성 (generator-core)
⑥ 결과 출력 → return code
"✨ Done!" 메시지 + Next steps 안내

4.2 단계별 호출 디테일

1
parseCliArgs(argv) args.ts

Node.js 표준의 util.parseArgs로 인자를 분석합니다. 문자열 배열 ["my-app", "--auth=local-json"] → 구조화된 객체 { projectName: "my-app", authMode: "local-json" }로 변환합니다.

{
  projectName: "my-app",
  authMode: "local-json",
  runInstall: undefined,    // --no-install이 없으면 undefined
  runGitInit: undefined,
  silent: undefined,
  ...
}

처리하는 flag: --auth, --no-i18n, --no-install, --no-git, --no-playwright, --silent, --verbose, --help, --version.

2
--help / --version 단축 경로

args.help가 true면 HELP_TEXT를 출력하고 exit 0. args.version이 true면 package.json의 version을 출력하고 exit 0. 여기서 빠지면 다음 단계 안 감.

if (args.help) {
  process.stdout.write(HELP_TEXT + "\n");
  return 0;  // ← exit 0으로 종료
}
3
logger + banner 준비 logger.ts, banner.ts

색상은 TTY(터미널)인지 + --silent 여부로 결정. 로그 레벨은 --verbose ? "debug" : --silent ? "silent" : "info".

const color = !args.silent && process.stdout.isTTY;
const level = args.verbose ? "debug" : args.silent ? "silent" : "info";

const logger = createLogger({
  out: (s) => process.stdout.write(s),
  err: (s) => process.stderr.write(s),
  level,
  color,
});

if (!args.silent) {
  process.stdout.write(renderBanner({ version: PKG.version, color }));
}
4
resolveScaffoldOptions(args) prompts.ts

인자에 빠진 값이 있으면 사용자에게 질문합니다. --auth 안 줬으면 → "Auth mode 고르세요", --no-i18n 안 줬으면 → "i18n 포함할까요?". 주의: --silent는 prompts를 자동 답하지 않습니다 — 누락하면 멈춤!

사용자가 Ctrl+C 누르면 AbortedError throw → exit 130.

5
scaffold(options) generator-core

드디어 실제 파일 생성. 이 함수가 9단계 작업을 수행합니다 — 다음 챕터의 주제입니다.

const result = await scaffold({
  projectName: resolved.projectName,
  targetDir,
  templateDir: findTemplateRoot(),
  authMode: resolved.authMode,
  ...
  packageManager: pm,
  logger,
});
6
결과 출력

성공이면 "✨ Done! Created my-app at /path/to/my-app" + Next steps 안내. 경고가 있으면 (post-install 일부 실패 등) 추가로 출력. 실패면 catch 블록에서 ScaffoldError를 적절한 exit code로 매핑.

4.3 에러 catch 블록

scaffold()가 throw하면 catch 블록이 받아 사용자에게 친절한 메시지 + 적절한 exit code를 만듭니다.

src/cli.ts catch 블록 
try {
  const result = await scaffold({ ... });
  ...
  return 0;
} catch (e) {
  if (e instanceof TargetExistsError) {
    logger.error(`Target directory not empty: ${e.path}`);
    return 1;  // 폴더 이미 있음
  }
  if (e instanceof ScaffoldError) {
    logger.error(`${e.code}: ${e.message}`);
    return e.code === "TemplateMissing" ? 2 : 3;
  }
  logger.error(`Unexpected error: ${(e as Error).message}`);
  return 99;  // 예상치 못한 에러
}

4.4 exit code 의미

code의미언제 발생
0✅ 성공scaffold 완료. (post-install 일부 실패는 warning이고 0)
1⚠️ TargetExists지정한 폴더가 이미 비어있지 않음
2⚠️ TemplateMissingtemplates/next-app-base/ 폴더가 없음 (재설치 필요)
3⚠️ 그 외 ScaffoldErrorInvalidProjectName, CopyFailed 등
99❌ 예상치 못한 예외버그. --verbose로 stack trace 확인
130⏸️ 사용자 중단Ctrl+C 또는 prompts에서 빠져나감

4.5 주요 부속 모듈 요약

cli.ts는 다음 5개 형제 모듈과 함께 일합니다.

args.ts

인자 파싱. parseCliArgs, HELP_TEXT export.

prompts.ts

대화형 질문. resolveScaffoldOptions, AbortedError export.

banner.ts

"▰▰▰ KMSF" 배너 렌더링. renderBanner export.

logger.ts

info/warn/error/step. ScaffoldLogger 인터페이스 구현.

cli.ts

위 4개를 묶고 scaffold()를 부르는 오케스트레이터.
