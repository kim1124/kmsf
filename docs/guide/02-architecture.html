모노레포 큰 그림

한 줄 요약

이 저장소 안에는 6종류의 워크스페이스가 있고, 각자 역할이 다르다. 우리가 주로 만지는 건 apps/kmsf/(reference 앱), packages/create-kmsf/(CLI), packages/generator-core/(엔진), templates/next-app-base/(붕어빵 틀) 4개다.

2.1 모노레포가 뭔가요?

회사 책장을 떠올려 보세요. 한 책장 안에 여러 권의 책이 꽂혀 있습니다. 각 책은 자기만의 표지/목차를 가지고 있지만, 같은 책장(저장소)에 있어서 옆 책을 쉽게 가져다 쓸 수 있습니다.

모노레포(Monorepo)도 똑같습니다. 한 git 저장소(=책장) 안에 여러 패키지(=책)가 들어있고, 서로 import해서 쓸 수 있는 구조입니다.

2.3 4개의 핵심 워크스페이스

이 4개만 알면 KMSF의 80%를 이해한 셈입니다. 나머지는 보조 역할입니다.

1
apps/kmsf/ reference 앱

실제로 npm run dev으로 띄울 수 있는 진짜 Next.js 앱입니다. 이 앱이 KMSF의 "공식 정답"이며, 모든 변경은 여기서 시작합니다.

2
packages/generator-core/ 엔진

"파일 복사", "{{token}} 치환", "auth 모드별 파일 제거" 같은 저수준 동작을 모아놓은 라이브러리입니다. 사용자는 직접 호출하지 않고 create-kmsf가 import해서 씁니다.

3
packages/create-kmsf/ CLI

사용자가 npx create-kmsf로 직접 실행하는 도구입니다. 인자 파싱, 대화형 질문, 배너 출력 같은 사용자 인터페이스를 담당합니다.

4
templates/next-app-base/ 붕어빵 틀

create-kmsf가 새 프로젝트를 만들 때 그대로 복사하는 정적 파일 트리입니다. apps/kmsf에서 sync 스크립트로 자동 생성합니다.

2.4 의존 관계 (누가 누구를 쓰는가)

화살표 방향: A → B는 "A가 B를 쓴다(import / copy / sync)"

2.5 신입에게 가장 헷갈리는 두 가지

⚠️ apps/kmsf와 templates/next-app-base는 거의 같은 내용

맞습니다. 의도적입니다. apps/kmsf는 "지금 바로 동작하는 앱"이고, templates/next-app-base는 "그걸 신규 프로젝트로 복사할 수 있게 약간 다듬은 정적 사본"입니다. 다만 templates는 수동 편집 금지 — sync 스크립트만 변경 권한이 있습니다.

⚠️ generator-core와 create-kmsf 둘 다 packages/ 안에 있는데?

맞습니다. 둘은 한 쌍의 라이브러리 + CLI입니다. generator-core는 "엔진"이고, create-kmsf는 그 엔진 위에 사용자 친화적 인터페이스를 입힌 도구입니다. ESLint(엔진) + ESLint CLI 같은 관계와 비슷합니다.
