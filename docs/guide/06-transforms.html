Chapter 6 / 10

transforms 4총사

한 줄 요약

copy로 일단 다 복사한 뒤, 4개의 transform이 결과물을 다듬는다: auth-mode (파일 제거 + sign-in 교체) → package-json (의존성 정리) → tokens (텍스트 치환) → env (.env.local 생성). 각자 1개의 일만 한다.

6.1 4총사의 위치

모두 packages/generator-core/src/transforms/ 폴더 안에 있습니다.

파일 트리 
packages/generator-core/src/transforms/
├── auth-mode.ts        ← 모드별 파일 제거 + sign-in 교체
├── env.ts              ← .env.example → .env.local
├── package-json.ts     ← name 치환 + 의존성 정리
└── tokens.ts           ← {{token}} 텍스트 치환

6.2 각자의 임무 한눈에

① auth-mode.ts

모드별 파일 제거. local-json이면 supabase 폴더 삭제. supabase면 local-json 파일 삭제. none이면 auth 전체 삭제.

② package-json.ts

name 치환 + supabase 의존성 정리. {{project_name}}을 실제 이름으로. local-json/none이면 @supabase/* deps 제거.

③ tokens.ts

텍스트 치환 엔진. README, .ts, .tsx, .json 안의 {{project_name}}을 일괄 치환.

④ env.ts

.env.local 생성기. .env.example을 복사하고 KMSF_AUTH_PROVIDER 설정. local-json은 32바이트 hex 시크릿 자동 생성.

6.3 자세히 보기 — auth-mode.ts

호출 시그니처

applyAuthMode(projectRoot: string, mode: AuthMode): Promise<void>

핵심 로직

const REMOVAL_MAP = {
  "local-json": [
    "src/lib/supabase",
    "src/lib/auth/google-identity.ts",
    "src/app/auth/callback",
  ],
  supabase: [
    "src/lib/auth/local-session.server.ts",
    "src/lib/auth/providers/local-json-auth-store.ts",
  ],
  none: [
    "src/lib/auth", "src/lib/supabase", "src/components/auth",
    "src/app/[locale]/(public)", "src/app/auth", ...
  ],
};

// 1. 모드에 따라 파일들 삭제
for (const rel of REMOVAL_MAP[mode]) {
  await rm(path.join(projectRoot, rel), { recursive: true, force: true });
}

// 2. local-json만: sign-in 페이지를 clean version으로 교체 (Q1)
if (mode === "local-json") {
  await writeLocalJsonSignIn(projectRoot);
}

📜 사연 있는 코드 — Q1 fixup

처음에는 sign-in 페이지의 Google OAuth 부분을 regex로 잘라내는 방식이었습니다. 그런데 실제 sign-in 페이지가 multi-line import를 쓰고, supabase 함수도 5군데에 흩어져 있어서 regex로는 깨끗이 못 잡았습니다. 결국 "sign-in 페이지를 통째로 새 파일로 교체"하는 방식으로 바꿨습니다 (commit 1b3100a).

6.4 자세히 보기 — package-json.ts

호출 시그니처

transformPackageJson(filePath, { projectName, authMode }): Promise<void>

핵심 로직

// 1. 이름 검증 (npm 규칙)
const validated = validateProjectName(options.projectName);

// 2. JSON 읽고 name 교체
const pkg = JSON.parse(await readFile(filePath));
pkg.name = validated;

// 3. supabase 모드가 아니면 supabase deps 제거
if (options.authMode !== "supabase") {
  delete pkg.dependencies["@supabase/ssr"];
  delete pkg.dependencies["@supabase/supabase-js"];
}

// 4. 다시 저장
await writeFile(filePath, JSON.stringify(pkg, null, 2) + "\n");

이름 규칙 (validateProjectName)

  • 1~214자
  • 첫 글자: a-z 또는 0-9
  • 나머지 글자: a-z 0-9 . _ - 만 허용
  • 대문자 / 공백 / 특수문자 → 거부 (npm 표준 규칙 간소화)

6.5 자세히 보기 — tokens.ts

호출 시그니처

substituteTokens(rootDir, { tokens, include }): Promise<{ filesModified }>

scaffold()가 부르는 형태

await substituteTokens(targetDir, {
  tokens: { project_name: options.projectName },
  include: ["**/*.json", "**/*.md", "**/*.ts", "**/*.tsx"],
});

치환 규칙

패턴: {{token_name}} (영숫자 + 밑줄). 등록되지 않은 토큰은 그대로 둠 (literal로 출력).

// 변환 전
{ "name": "{{project_name}}" }

// 변환 후
{ "name": "my-app" }

🐛 버그 한 토막

처음 spec의 compilePattern은 **/*.json이 root-level 파일(예: package.json)을 매칭 못 했습니다. 그래서 token 치환이 root-level 파일을 건너뛰는 버그가 있었습니다. 지금은 tokenizer 방식으로 **/를 (?:.*/)?로 바꿔서 root까지 포함합니다.

6.6 자세히 보기 — env.ts

호출 시그니처

generateEnvLocal(projectRoot, { authMode }): Promise<{ created, skippedReason? }>

로직 흐름

  1. .env.example이 있는지 확인. 없으면 throw.
  2. .env.local이 이미 있으면 skip하고 종료.
  3. .env.example을 읽음.
  4. KMSF_AUTH_PROVIDER=... 라인을 사용자가 고른 모드로 교체.
  5. local-json 모드면 추가로 KMSF_LOCAL_AUTH_SESSION_SECRET을 32바이트 랜덤 hex로 자동 생성.
  6. .env.local에 저장.

local-json 모드 출력 예시

# Application
NEXT_PUBLIC_APP_URL=http://127.0.0.1:3000

# Auth provider mode
KMSF_AUTH_PROVIDER=local-json

# Supabase (local-json이라 비워둠)
NEXT_PUBLIC_SUPABASE_URL=
NEXT_PUBLIC_SUPABASE_ANON_KEY=

# Local JSON auth — 자동 생성됨!
KMSF_LOCAL_AUTH_DB_PATH=./.local/auth.db.json
KMSF_LOCAL_AUTH_SESSION_SECRET=829c8285ef13...b7e5a

6.7 호출 순서가 중요한 이유

4개 transform은 scaffold() 안에서 정해진 순서로 호출됩니다. 순서를 바꾸면 안 되는 이유를 예시로 봅시다.

순서왜 그래야 하는가
auth-mode 먼저 제거할 파일에 token 치환을 적용하면 헛수고. 일단 정리부터.
package-json 다음 auth-mode가 supabase 폴더 지웠으니, package.json의 supabase deps도 제거해야 일관됨.
tokens 다음 package.json의 name 필드가 이미 깨끗해진 상태에서 README 등 다른 파일 토큰 치환.
env 마지막 모든 코드 변환이 끝난 후 .env.local만 생성하면 됨. 다른 transform이 .env.local을 건드릴 일 없음.

6.8 호출 계통도

scaffold()가 4개 transform을 step 4~7에서 순서대로 호출한다.

6.9 이번 챕터의 핵심

📌 한 모듈은 한 가지 일만

4개 transform 모두 한 줄로 설명되는 단일 책임을 가집니다. 새 변환이 필요하면 기존 모듈에 끼워넣지 말고 새 파일을 만드는 게 좋습니다 (스타일 컨벤션).
