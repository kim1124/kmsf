## 02:20 KMSF Remaining UI Auth Validation Hardening

### 작업 요약

- 헤더의 `KMSF` 브랜드 클릭 시 대시보드로 이동하도록 변경했다.
- 현재 경로에 따라 좌측 Aside와 모바일 하단 네비게이션의 활성 상태가 정확히 유지되도록 정리했다.
- 헤더와 동일 높이의 푸터를 추가하고, 서버 시각 시드값을 기준으로 초 단위 갱신 시계를 좌측 `ml-[10px]` 위치에 노출했다.
- 보호 레이아웃 전환 시 즉시 로딩 UI를 노출해 설정 페이지 등에서 느껴지던 빈 화면 지연을 줄였다.
- 개발 서버 재기동 후에도 세션이 남아 보이던 문제를 앱 세션 쿠키와 런타임 ID 검사로 차단했다.
- 마우스, 키보드, 휠, 터치 이벤트가 1시간 동안 없으면 자동 로그아웃되도록 세션 모니터를 추가했다.
- 인증/프로필/초기 관리자 폼에 실시간 클라이언트 유효성 검사와 입력 sanitize 흐름을 연결했다.
- 텍스트 입력값은 `isomorphic-dompurify` 기반 sanitize를 거친 뒤 검증하도록 통일했다.
- Superpowers 실사용 흐름을 이번 작업 기준으로 재정리했다.

- 폼 입력 중 실시간 클라이언트 검증과 `isomorphic-dompurify` 기반 sanitize가 unit test와 실제 렌더 경로에 모두 연결됐다.
- 개발 런타임이 바뀌면 기존 앱 세션 쿠키를 만료로 처리하므로, 개발 서버가 재시작된 뒤에도 보호 화면이 유지되지 않는다.

### Superpowers 사용 방법 검토

- 이번 작업에서는 `brainstorming`으로 요구사항을 분해하고, `writing-plans`로 구현 순서를 고정한 뒤, `test-driven-development`로 실패 테스트를 먼저 추가하고, 마지막에 `verification-before-completion`으로 완료 게이트를 강제하는 방식이 가장 실용적이었다.
- 구조 변경이 아니라 기능 변경이 중심일 때는 `test-driven-development`의 효과가 크다. 예를 들어 이번 작업에서는 세션 만료, 활성 메뉴, 푸터 시계, 실시간 검증처럼 기대 결과가 명확한 항목에 바로 적용할 수 있었다.
- 반대로 경로 이동이나 workspace 재구성처럼 파일 배치 비중이 큰 작업은 `writing-plans`와 `verification-before-completion`을 우선 두고, 필요한 부분만 TDD로 보강하는 편이 안정적이다.

### 잔여 이슈 및 후속 작업

- 1시간 유휴 로그아웃은 쿠키 만료 계산과 브라우저 이벤트 경로로 검증했지만, 실제 1시간 대기 방식의 장시간 E2E는 수행하지 않았다.
- 입력 sanitize는 현재 텍스트 기반 입력 경로를 중심으로 `isomorphic-dompurify`를 적용했다. 향후 리치 텍스트나 HTML 입력 기능이 추가되면 정책을 분리한 전용 sanitizer 계층을 별도로 두는 편이 더 안전하다.
