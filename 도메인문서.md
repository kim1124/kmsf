## 0. 한 줄 요약

KMSF는 Next.js 16 기반 admin dashboard **모노레포**다. `apps/kmsf`가 reference 구현이고, `npx create-kmsf`로 이 템플릿을 단독 Next.js 앱으로 scaffold할 수 있다.

## 1. 도메인 모델 — 무엇이 무엇인가

### 1.1 워크스페이스 구성

| 워크스페이스 | 역할 | 산출물 |
|---|---|---|
| `apps/kmsf` | reference 앱 (실제 운영 가능한 admin dashboard) | Next.js 빌드 |
| `packages/generator-core` (`@kmsf/generator-core`) | scaffold 엔진 (copy, transforms, post-install) | npm package, dist/ 빌드 필요 |
| `packages/create-kmsf` (`create-kmsf`) | npx 진입 CLI | `bin/create-kmsf.js` + dist/ |
| `packages/charts`, `packages/gridstack` | 재사용 UI 패키지 | (현재는 보일러플레이트에서 제외) |
| `templates/next-app-base` | scaffold 시 복사되는 정적 트리 | `scripts/sync-template.sh`로 생성 |
| `examples/basic-dashboard` | 패키지 소비 예제 | (future) |
| `scripts/sync-template.sh` | apps/kmsf → templates 동기화 | shell script |
| `.github/workflows/sync-template-check.yml` | drift 검출 + 통합 테스트 | CI gate |

### 1.2 데이터 흐름 (scaffold 1회 실행 시)

```
사용자: npx create-kmsf my-app --auth=local-json
  ↓
create-kmsf bin → dist/cli.js
  ↓
cli.ts: parseArgs → resolveScaffoldOptions(prompts) → scaffold()
  ↓
generator-core scaffold():
  1. ensureTemplateExists(templateDir)            # TemplateMissing 검출
  2. isEmptyOrMissing(targetDir)                  # TargetExists 검출
  3. copyDir(templateDir → targetDir, EXCLUDE)    # 파일 복사
  4. applyAuthMode(authMode)                      # 모드별 파일 제거 + sign-in 교체
  5. transformPackageJson(name, authMode)         # name 치환 + dep 정리
  6. substituteTokens({{project_name}}, ...)      # 텍스트 토큰 치환
  7. generateEnvLocal(authMode)                   # .env.example → .env.local
  8. (optional) runInstall(packageManager)        # npm/pnpm/yarn install
  9. (optional) runPlaywrightInstall()            # Playwright 브라우저
 10. (optional) runGitInit()                      # git init + initial commit
  ↓
ScaffoldResult { projectRoot, filesCopied, durationMs, warnings[] }
```

### 1.3 Auth 모드 3종

| 모드 | 의미 | 보존 파일 | 제거 파일 |
|---|---|---|---|
| `local-json` | 파일 기반 auth (`./.local/auth.db.json`) | `local-session.server.ts`, `providers/local-json-auth-store.ts`, `components/auth/*` | `lib/supabase/`, `google-identity.ts`, `auth/callback/`. **추가**: sign-in/page.tsx를 Google-free 버전으로 **교체** |
| `supabase` | Supabase Auth (Google OAuth 포함) | `lib/supabase/*`, `google-identity.ts`, `auth/callback/` | `local-session.server.ts`, `local-json-auth-store.ts` |
| `none` | auth 코드 전부 제거 | (없음) | `lib/auth/`, `lib/supabase/`, `components/auth/`, `[locale]/(public)/`, `auth/`, `api/session/`, `setup/` |

## 3. 불변 규칙 (IMMUTABLE RULES)

> 이 규칙들은 **변경 시 새 ADR 필요**. 단순 PR로 깰 수 없다.

### 3.1 코드 흐름 불변 규칙

1. **scaffold()의 단계 순서는 위 §1.2의 순서를 따른다.** copy → applyAuthMode → transformPackageJson → substituteTokens → generateEnvLocal → post-install. 순서를 바꾸면 token 치환이 supabase가 제거된 자리에 적용되거나, 반대로 토큰이 치환된 코드가 다시 제거되는 등의 비대칭 버그가 발생한다.

2. **CopyDir의 EXCLUDE는 POSIX path 기준 glob.** Windows 호환성은 향후 검증 필요(현재는 macOS/Linux만 지원).

3. **applyAuthMode는 idempotent.** 같은 모드를 두 번 실행해도 결과가 같다 (이미 제거된 파일은 ENOENT가 무시됨). 새 모드를 추가할 때도 이 성질을 유지하라.

4. **post-install 실패는 throw 하지 않는다.** 반드시 `{ success: false, reason }`을 반환하고 caller가 `result.warnings`로 전달한다 (D-15).

5. **error type discriminator는 `code` 필드.** 새 에러 타입을 추가할 때 `ScaffoldErrorCode` union에 추가하고, `ScaffoldError` 베이스를 상속한다. CLI는 `code`로 exit code를 매핑한다 (`TemplateMissing`=2, `TargetExists`=1, 기타 ScaffoldError=3, unexpected=99).

### 3.2 sync-template.sh 불변 규칙

6. **rsync EXCLUDE에 `[locale]` 같은 대괄호가 들어갈 때는 backslash escape.** `[locale]`은 character class로 해석되어 매칭 안 됨. 반드시 `'src/app/\[locale\]/...'` 형태로 작성. (B1 버그)

7. **sync 스크립트는 한 방향이다 — `apps/kmsf` → `templates/next-app-base`.** 절대 반대로 적지 말 것. 사용자 지시 없이 templates를 수동 편집하면 다음 sync가 그것을 덮어쓴다.

8. **CI의 `--check` 모드는 main에 들어가는 모든 PR에서 통과해야 한다.** drift는 두 트리가 갈라지는 첫 신호 — 무시하면 며칠 안에 templates가 의미를 잃는다.

### 3.3 generator-core 빌드 불변 규칙

9. **`packages/generator-core`와 `packages/create-kmsf`는 모두 `dist/` 빌드가 있어야 런타임에 작동.** Node ESM은 `.ts`를 직접 import 못 한다. 만약 `package.json`의 `exports`를 `./src/...`로 되돌리면 vitest는 통과하지만 production CLI는 `ERR_MODULE_NOT_FOUND`로 깨진다 (commit `9ae0381`의 fix는 절대 되돌리지 말 것).

10. **모든 internal `import`는 `.js` 확장자.** TypeScript NodeNext 모듈 해상도 규칙. `from "./foo.ts"`나 `from "./foo"`는 컴파일러는 통과해도 emit된 코드가 깨진다.

### 3.4 Auth 모드 불변 규칙

11. **local-json 모드의 sign-in 페이지는 wholesale 교체로 처리한다 (Q1).** regex 수술은 신뢰할 수 없음 (multi-line import, intertwined supabase 함수 등). 새 auth 관련 파일을 sign-in에 추가하면 `LOCAL_JSON_SIGNIN_PAGE` 상수도 함께 업데이트.

12. **`REMOVAL_MAP`의 path는 모두 project root 기준 relative POSIX path.** Windows에서 동작하려면 path.join이 처리하므로 평가/조작 시 normalize 안 됨. 새 mode를 추가하면 `REMOVAL_MAP`을 갱신하고 `applyAuthMode`의 conditional 분기도 검토하라.

### 3.5 CLI 인터페이스 불변 규칙

13. **`--silent`는 prompt를 자동 응답하지 않는다.** CI에서는 모든 인터랙티브 옵션을 flag로 제공해야 한다 (예: `--auth=local-json --no-i18n --no-install --no-git --no-playwright --silent`). 이 동작은 D-11에 의해 의도적이다. 변경 시 integration test의 spawn 인자 목록도 갱신.

14. **exit code 매핑은 spec §7과 cli.ts의 catch 블록에서 단일 진실 소스다.** TargetExists=1, TemplateMissing=2, 그 외 ScaffoldError=3, AbortedError=130, 그 외 = 99.

## 4. AI Agent 협업 가이드

> AI에게 작업을 시킬 때 반드시 전달해야 할 컨텍스트.

### 4.1 일반 원칙

- **이 저장소는 AGENTS.md / CLAUDE.md / docs/codex/ 기반의 instruction layer가 있다.** 처음 들어오는 agent는 `AGENTS.md`(루트), `apps/kmsf/AGENTS.md`, `apps/kmsf/tests/AGENTS.md`를 먼저 읽어야 한다.
- **branch는 `codex-kmsf-auth-local-json-delivery`이며, main이 아니다.** push는 사용자 승인 후에만.
- **테스트 라우팅**은 `apps/kmsf/tests/AGENTS.md` 참고. 무작정 e2e 돌리지 말 것.

### 4.2 변경 유형별 워크플로

| 변경 | 워크플로 |
|---|---|
| `apps/kmsf` 변경 (UI/auth/routing) | `bash scripts/sync-template.sh`로 templates 갱신 → 양쪽 commit |
| `packages/generator-core` 변경 | TDD (test 먼저). 빌드 후 `node packages/create-kmsf/bin/create-kmsf.js --version` 검증 |
| `packages/create-kmsf` 변경 | TDD + build + `--help`/`--version` smoke |
| 새 auth 모드 추가 | `AuthMode` union + `REMOVAL_MAP` + (필요 시) `applyAuthMode` 분기 + e2e 테스트 |
| 새 template 추가 | `templates/<id>/` 디렉터리 + `TEMPLATE_CATALOG` 항목 + sync 스크립트 (또는 별도 sync) |

### 4.3 흔한 함정

1. **`.env.example`은 root `.gitignore`의 `.env*` 규칙으로 차단된다.** templates용으로는 `!templates/**/.env.example` negation이 추가되어 있다. 새 auth 모드를 위해 별도 .env.example을 추가하면 negation 패턴도 업데이트.

2. **`packages/create-kmsf/src/cli.ts`의 placeholder는 Phase 5.1에서 교체됐다.** 단순 빈 파일로 보이는 것이 있다면 그것은 placeholder가 아니라 의도된 stub일 수 있다. **놀라기 전에 git log를 확인.**

3. **integration test의 spawn 인자에 `--no-i18n`이 빠지면 hang.** D-11 / 불변 규칙 13 참고.

4. **generator-core의 `compilePattern`은 plan spec과 다른 구현이 두 곳에 있다** (`copy.ts`, `transforms/tokens.ts`). 이는 spec 버그를 우회한 의도적인 divergence. 두 파일을 통합하려면 먼저 root-level file 매칭 + `dir/**`의 directory prune 두 동작을 모두 보존하는지 확인 (regression test 작성).

### 4.4 코드 안에서 AI 주석을 만나면

- `// AI-NOTE:` 블록은 변경 시 검토가 필요한 invariant 또는 trade-off의 출처를 알려준다. 무시 금지.
- 주석에 "IMMUTABLE RULES" 섹션이 있으면 위의 §3과 매핑된다. 변경하려면 도메인문서.md도 함께 업데이트.
